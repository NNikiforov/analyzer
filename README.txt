Задача: есть некое хранилище текстовых файлов (без вложенных директорий),
необходимо написать программу для вычисления похожести файлов в этом хранилище.
Файлы сравниваются каждый с каждым и для каждой пары выводится некий
коэффициент похожести. 

Похожесть вычисляется по количествуодинаковых слов которые встречаются
одинаковое кол-во раз в сравниваемых файлах.


Для решения данной задачи я в главном потоке считал имена всех файлов, лежащих
в указанном каталоге. Затем в главном потоке делегировал задачу отображения
содержимого файла на Set<Entry<String, Integer>> (множество пар "слово" - "количество")
нескольким параллельным потокам. После завершения вычисления отображения, объект файла 
с отображением попадал в коллекцию. А главный поток выделял новые задания по определению
похожести двух файлов для потоков. Благодаря промежуточному звену в виде коллекции не было
необходимости ждать, пока все файлы получат своё отображение. Т. е. если существуют свободные
потоки (в моём приложении их количество ограничено количеством ядер на компьютере) и все задачи
по отображению выполненны, то поток может выполнять анализ похожести двух уже готовых файлов.
